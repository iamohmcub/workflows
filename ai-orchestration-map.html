AI Engineering Orchestration Map

How `.ai` and `AGENTS.md` organize provider-neutral AI work across repo groups, repos, roles, runtime events, SDLC gates, logs, and impact reports.

Repo Group -> Repo -> Role

Repo groups coordinate many company/domain repos. Repos run SDLC phases. Roles execute expert work under the same global rules.

Layer 1

Repo Group

Workspace for many companies, repos, products, services, or domains.

  • Owns shared rules
  • Tracks cross-repo impact
  • Escalates unresolved conflicts
Layer 2

Repo

Company or domain unit that runs the full software development cycle.

  • Owns phase logs
  • Owns repo decisions
  • May declare child repos
Layer 3

Role

Domain expert agent with character, skills, responsibilities, hooks, and rules.

  • Owns specialist output
  • Creates handoff notes
  • Reports impact outside scope

New Project Install

Default install is minimal and chat-first. Commands and local scripts are optional helpers, not the required user experience.

Default: Minimal

  • `npx @iamohmcub/ai-orchestration init` copies `.ai/` and `AGENTS.md`.
  • Works for Node, Go, Python, mobile, infra, documentation, or mixed repos.
  • The AI worker reads files directly and maps natural chat to workflow events.
  • Project-specific reality lives in `.ai/project.yml`, `.ai/workspace`, and `.ai/runtime`.

Optional: Commands

  • `--commands` also copies `COMMANDS.md`.
  • `--with-local-cli --with-package-scripts` adds local scripts for teams that want repo-local commands.
  • Central library repo keeps `.github` validation; installed projects do not need it by default.
  • The same `.ai` files remain the source of truth with or without an adapter.

Portable Runtime

`.ai` is the source of truth. Adapters, plugins, CI, MCP, and local LLM runners are optional engines that read the same files.

Stable Standard

  • `AGENTS.md` and `.ai/manifest.yml` define load order.
  • `worker.contract.yml` defines provider-neutral worker behavior.
  • `routing.matrix.yml` decides who does, reviews, and must be notified.
  • `parallel.delivery.yml` keeps MVP lanes independent unless shared contracts collide.

Changing Runtime

  • `project.yml` stores repo identity, active roles, and current phase.
  • `runtime/state.yml` stores current work item, lane, queue, and evidence roots.
  • `runtime/queue` holds pending provider-neutral events.
  • `runtime/events` holds completed or archived events.

Parallel MVP Lanes

Product, design, engineering, QA, release, and learning can work on different MVPs at the same time once handoffs lock the contract for each work item.

Allowed Flow

  • Business hands off `mvp-1` to design or engineering.
  • Engineering builds `mvp-1` while business starts `mvp-2` discovery.
  • QA tests `mvp-1` while frontend/backend continue implementation tasks.
  • Monitoring measures released MVPs while product plans the next iteration.

Guardrails

  • Every parallel log names work item, lane, upstream handoff, and dependencies.
  • Handoff locks scope, AC, design, API, data, and release expectation.
  • Changing a locked handoff needs an impact report.
  • Shared security, compliance, environment, API, data, or customer dependencies can block multiple lanes.

Branch, PR/MR, Review, Merge

Parallel work is isolated by branch and integrated through provider-neutral change requests: GitHub PR, GitLab MR, or equivalent.

Branch and PR/MR

  • Create a dedicated branch for feature, fix, hotfix, chore, docs, test, refactor, or workflow work.
  • Branch names include the feature or work item when known.
  • Open a PR/MR when ready for review or early feedback.
  • PR/MR links phase log, evidence, tests, impact reports, handoffs, and decisions.

Review and Merge

  • Reviewers come from `.ai/global/routing.matrix.yml`.
  • No self-approval as the only reviewer.
  • Merge requires approval, green CI, tests, lint, secret scan, and impact acknowledgement.
  • After merge, update the phase log and trigger downstream handoff when another lane is unlocked.

Conflict Handling

  • Identify conflicted files, file owners, domain owners, branch owners, feature/work item, and impacted contracts.
  • The role owning the conflicted code decides domain behavior; orchestrator resolves process conflict.
  • Do not choose one side silently. Record the decision and preserve both intents when possible.
  • Rerun impacted tests and create or update an impact report when API, data, UI, infra, security, release, or customer expectations change.

Logs and Impact Reports

The goal is not just to finish work. The goal is to finish each phase with evidence and report anything that affects another role or repo.

Phase starts Create phase log from `.ai/templates/phase-log.md`.
Role works Use role character, skills, responsibilities, hooks, and rules.
Impact check If another role or repo is affected, create impact report.
Gate review Check DoD item by item. No bulk approval.
Handoff Write handoff note and link evidence before next phase.

Non-Negotiables

These rules apply across every repo group, repo, role, phase, and emergency path.

Delivery

  • Every phase must end with a phase log.
  • DoD is checked item by item.
  • DEV -> STAGING -> UAT -> PROD is the default flow.
  • CI, peer review, and secret scanning are required.

Coordination

  • Cross-role impact requires a report.
  • Cross-repo impact requires a group report.
  • Orchestrator resolves unresolved conflicts.
  • PR/MR approval and merge require reviewer, check, and evidence gates.
  • P0 and P1 incidents require post-mortems within 24 hours.
